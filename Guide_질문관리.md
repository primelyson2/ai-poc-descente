# 질문관리 가이드 — Select AI Test - Table list

`Select AI Test - Table list` 화면에서 **질문과 조회할 컬럼을 관리하는 방식**을 설명합니다. 화면 상단의 **질문관리방식** 라디오로 두 가지 중 하나를 선택합니다.

| 방식 | 저장 위치 | 공유 | 용도 |
|---|---|---|---|
| **local storage** | 브라우저(사용자별) | ✕ 개인 전용 | 각자 자주 쓰는 질문/컬럼을 빠르게 저장 |
| **DB** | Oracle DB 테이블(접속 스키마) | ✅ 같은 DB 사용자 공유 | 검증된 좋은 질문·조회컬럼 세트를 공유·재사용 |

> 선택한 방식은 **접속 DB별로 기억**되어, 다음 방문 시 자동으로 복원됩니다.

---

## 1. local storage 방식 (기본)

기존과 동일합니다. 화면의 **질문 / 조회할 컬럼 / 정렬기준** 입력란을 직접 쓰고, 각 입력란의 **저장 콤보(제목 + 추가/수정/삭제 + 저장된 목록)** 로 개인 브라우저(localStorage)에 저장·재사용합니다.

- 저장 데이터는 **그 브라우저·그 사용자**에게만 남습니다(다른 사람과 공유되지 않음).
- DB에 아무것도 만들지 않습니다.

---

## 2. DB 방식 (공유)

질문과 조회컬럼을 **Oracle DB 테이블에 등록**해 두고, **검색·선택**해서 쓰는 방식입니다. 같은 DB에 접속하는 모든 사용자가 동일한 질문·컬럼 목록을 공유합니다.

DB 방식을 선택하면 화면의 localStorage 입력란(질문/조회컬럼/정렬기준)이 사라지고, **[질문 선택]** 버튼과 선택 결과 요약이 표시됩니다.

### 2.1 전체 흐름

```
[질문 선택] 버튼
   │
   ▼
질문 선택 빌더 (질문 + 조회할 컬럼)
   ├─ [질문 검색] ─▶ 질문 검색 팝업 (부분일치 검색 / + 새 질문 등록)
   └─ [컬럼 선택] ─▶ 컬럼 선택 팝업 (질문별 컬럼 / 모든 조회칼럼 / + 컬럼 추가 / 관련성 평가)
   │
   ▼
[선택완료] ─▶ 화면의 질문·조회할 컬럼에 반영 ─▶ [실행]
```

### 2.2 질문 선택 (검색 · 등록)

**[질문 검색]** 을 누르면 질문 검색 팝업이 뜹니다.

- **부분일치 검색**: 검색어를 입력하면 질문 내용에 그 단어가 포함된 질문을 목록으로 보여줍니다(text like). 행을 클릭하면 선택됩니다.
- **새 질문 등록**: 검색란에 새 질문 내용을 적고 **[+ 새 질문 등록]** 을 누르면 DB에 저장되고 곧바로 선택됩니다. (테이블이 비어 있어도 여기서 처음 등록할 수 있습니다.)

> 빌더의 질문 입력란에 **직접 타이핑**한 뒤 [선택완료]/[컬럼 선택]을 하면, 그 질문이 **새 질문으로 DB에 저장**됩니다(추가 + 선택).

### 2.3 컬럼 선택 (질문별 · 전체 · 추가)

**[컬럼 선택]** 을 누르면 컬럼 선택 팝업이 뜹니다.

- **질문별 컬럼(기본)**: 현재 선택한 질문에 연결된 조회 컬럼만 체크박스로 보여줍니다.
- **모든 조회칼럼**: 체크박스를 켜면 **질문과 무관하게 DB에 등록된 모든 조회 컬럼**을 보여줍니다. 여기서 필요한 컬럼을 골라 담을 수 있습니다.
- **컬럼 추가**: 새 컬럼명을 입력하고 **[추가]** 하면 현재 질문에 연결되어 DB에 저장됩니다.

### 2.4 관련성 평가 (AI)

**"모든 조회칼럼"** 모드에서 컬럼을 선택하면 **[관련성 평가]** 버튼이 활성화됩니다. 누르면 AI가 **선택한 컬럼이 질문에 답하기에 적절한 조회 컬럼인지** 평가합니다.

- 평가에는 **현재 Chat설정의 AI Profile** 과 그 프로파일의 **스키마 컨텍스트(테이블/컬럼 Comment·Annotation 포함)** 가 사용됩니다.
- 결과는 **컬럼별 판정(관련 / 무관)과 사유**, 그리고 총평으로 표시됩니다.

예시:

| 컬럼 | 판정 | 사유 |
|---|---|---|
| 브랜드명 | 관련 | '브랜드별' 매출을 요구하므로 그룹화·식별에 필요 |
| 판매금액 | 관련 | '매출' 상위를 요구하므로 집계·비교에 필요 |
| 고객주소 | 무관 | 브랜드별 매출 순위 분석에는 불필요 |

> 평가는 **선택을 강제하지 않습니다.** 참고용 근거이며, 최종 선택은 사용자가 합니다.

### 2.5 선택완료 → 실행

**[선택완료]** 를 누르면 선택한 질문과 조회할 컬럼이 화면에 반영됩니다. 이후 **[실행]** 을 누르면 local storage 방식과 동일하게 SQL 생성·실행이 진행됩니다. (DB 방식에는 정렬기준이 없습니다.)

## 3. DB 테이블 구조

DB 방식은 접속 스키마에 아래 두 테이블을 **자동 생성**해 사용합니다(최초 사용 시 `CREATE TABLE IF NOT EXISTS` 로 멱등 생성).

### T_NL2SQL_QUESTION — 질문
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `ID` | NUMBER (IDENTITY, PK) | 질문 식별자 |
| `QUESTION` | CLOB | 질문 내용 |
| `QUESTION_VECTOR` | VECTOR | 질문 임베딩 — **유사 질문 검색용(선택, Oracle 23ai)**. 화면 자동생성에는 포함되지 않으며, 쓰려면 관리자가 추가·적재(§3.1) |
| `CREATED_AT` | TIMESTAMP | 등록 시각 |

### T_NL2SQL_COLUMN — 조회 컬럼
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `ID` | NUMBER (IDENTITY, PK) | 컬럼 식별자 |
| `QUESTION_ID` | NUMBER | 연결된 질문(`T_NL2SQL_QUESTION.ID`) |
| `COLUMN_NAME` | VARCHAR2(400) | 조회 컬럼명 |
| `CREATED_AT` | TIMESTAMP | 등록 시각 |

- **질문 : 컬럼 = 1 : N** — 한 질문에 여러 조회 컬럼을 연결합니다.
- **"모든 조회칼럼"** 은 `T_NL2SQL_COLUMN` 전체의 **distinct 컬럼명**을 보여줍니다(질문 무관).

### T_NL2SQL_SIMILAR — 유사 질문 *(선택, 유사 질문 검색용)*
`T_NL2SQL_QUESTION` 의 **하위 테이블**로, 하나의 대표 질문(`QUESTION_ID`)에 **여러 유사(대체 표현) 질문과 그 임베딩 벡터**를 연결해 관리합니다. 사용자가 표현을 달리 입력해도 유사질문 벡터에 매칭되면 **원본 대표 질문으로 매핑**할 수 있습니다.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `ID` | NUMBER (IDENTITY, PK) | 유사 질문 식별자 |
| `QUESTION_ID` | NUMBER | 원본 대표 질문(`T_NL2SQL_QUESTION.ID`) |
| `SIMILAR_QUESTION` | CLOB | 유사(대체 표현) 질문 내용 |
| `SIMILAR_VECTOR` | VECTOR | 유사 질문 임베딩(Oracle 23ai) |
| `CREATED_AT` | TIMESTAMP | 등록 시각 |

- **질문 : 유사질문 = 1 : N** — 한 대표 질문에 여러 표현의 유사 질문을 연결합니다.
- 앱 자동생성에는 **포함되지 않는 선택 테이블**입니다(23ai `VECTOR` 필요). 아래 생성 스크립트로 관리자가 만들어 사용합니다. 임베딩 적재·검색은 §3.1.

### 생성 스크립트

화면에서 DB 방식을 처음 사용하면 두 테이블이 **자동 생성**됩니다(이미 있으면 건너뜀). 다만 앱의 자동생성은 `QUESTION_VECTOR` 를 **만들지 않으므로**, 유사 질문 검색까지 쓰려면 아래 **전체 스크립트로 관리자가 미리 생성**하는 것을 권장합니다(`CREATE TABLE IF NOT EXISTS` 라 앱이 그 위에 다시 만들지 않습니다).

```sql
-- 1. 질문 테이블 (질문 임베딩 벡터 컬럼 포함 — Oracle 23ai)
CREATE TABLE IF NOT EXISTS T_NL2SQL_QUESTION (
  id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  question        CLOB,
  question_vector VECTOR,          -- 질문 임베딩 (유사 질문 검색용 · 적재는 §3.1)
  created_at      TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- 2. 조회 컬럼 테이블 (질문에 1:N 연결)
CREATE TABLE IF NOT EXISTS T_NL2SQL_COLUMN (
  id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  question_id NUMBER,                 -- T_NL2SQL_QUESTION.ID 참조(논리 FK)
  column_name VARCHAR2(400),
  created_at  TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- 3. 유사 질문 테이블 (대표 질문에 1:N 연결 · 유사질문 + 임베딩 벡터 — Oracle 23ai)
CREATE TABLE IF NOT EXISTS T_NL2SQL_SIMILAR (
  id               NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  question_id      NUMBER,            -- T_NL2SQL_QUESTION.ID 참조(논리 FK)
  similar_question CLOB,              -- 유사(대체 표현) 질문
  similar_vector   VECTOR,            -- 유사 질문 임베딩 (적재는 §3.1)
  created_at       TIMESTAMP DEFAULT SYSTIMESTAMP
);
```

> - `CREATE TABLE IF NOT EXISTS` 는 Oracle 23ai 문법입니다. 하위 버전이면 `IF NOT EXISTS` 를 빼고 실행하세요(이미 있으면 `ORA-00955`).
> - `GENERATED BY DEFAULT AS IDENTITY` 로 `ID` 가 자동 채번됩니다(별도 시퀀스 불필요).
> - `QUESTION_ID` 는 논리적 FK 입니다(제약조건 없이 애플리케이션에서 연결). 무결성 제약이 필요하면 `FOREIGN KEY (question_id) REFERENCES T_NL2SQL_QUESTION(id)` 를 추가할 수 있습니다.
> - 테이블 생성에는 접속 계정의 **`CREATE TABLE`** 권한이 필요합니다.
> - `QUESTION_VECTOR`(질문 임베딩)는 **Oracle 23ai `VECTOR` 타입**입니다. 앱의 자동생성에는 없으니, 위 스크립트로 미리 만들거나 기존 테이블이면 `ALTER TABLE T_NL2SQL_QUESTION ADD (question_vector VECTOR)` 로 추가하세요. 임베딩 적재·검색은 §3.1 참고.
> - `T_NL2SQL_SIMILAR`(유사 질문) 역시 **선택 테이블**로 앱이 자동생성하지 않습니다. `QUESTION_ID` 로 `T_NL2SQL_QUESTION` 에 논리 FK 로 연결되며, 대표 질문 삭제 시 이 테이블의 연결 행도 함께 정리해야 합니다.

## 3.1 질문 임베딩 벡터 컬럼 (유사 질문 검색)

현재 [질문 검색] 은 **텍스트 부분일치(text like)** 로 동작합니다. 표현이 다르면(예: "브랜드별 매출" vs "브랜드 기준 판매액") 못 찾을 수 있습니다. 질문을 **임베딩 벡터**로 저장해 두면 **의미가 비슷한 질문**을 벡터 유사도로 찾을 수 있습니다(Oracle 23ai `VECTOR` 타입).

> 이 기능은 **선택적 확장**입니다. 화면 자동생성에는 없으므로, 아래를 **관리자가 접속 스키마에서 직접 실행**해 컬럼을 추가하고 임베딩을 채워야 합니다.

### 3.1.1 벡터 컬럼 추가

```sql
-- 기존 T_NL2SQL_QUESTION 에 임베딩 컬럼 추가 (23ai)
ALTER TABLE T_NL2SQL_QUESTION ADD (question_vector VECTOR);
```

새로 만든다면 처음부터 포함할 수 있습니다.

```sql
CREATE TABLE IF NOT EXISTS T_NL2SQL_QUESTION (
  id              NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  question        CLOB,
  question_vector VECTOR,          -- 질문 임베딩 (유사 질문 검색용, 선택)
  created_at      TIMESTAMP DEFAULT SYSTIMESTAMP
);
```

> `VECTOR` 는 차원·형식을 지정하지 않으면 **어떤 임베딩이든** 담을 수 있습니다. 특정 모델로 고정하려면 `VECTOR(1024, FLOAT32)` 처럼 (차원, 형식)을 명시하세요 — 이때 적재하는 임베딩의 차원과 반드시 일치해야 합니다.

### 3.1.2 임베딩 생성·적재

임베딩을 만드는 방법은 크게 두 가지입니다.

**(A) DB 내장 ONNX 모델 — `VECTOR_EMBEDDING`**
임베딩 모델(ONNX)을 DB 에 한 번 적재해 두고, DB 안에서 바로 임베딩합니다(외부 호출 없음).

```sql
-- (최초 1회) 관리자: ONNX 임베딩 모델을 DB 에 적재
-- 모델 파일은 DBMS_CLOUD.GET_OBJECT 등으로 디렉터리에 내려받아 사용
BEGIN
  DBMS_VECTOR.LOAD_ONNX_MODEL(
    directory  => 'DATA_PUMP_DIR',
    file_name  => 'all_MiniLM_L12_v2.onnx',
    model_name => 'DOC_MODEL');
END;
/

-- 기존 질문 임베딩 일괄 채우기
UPDATE T_NL2SQL_QUESTION
   SET question_vector = VECTOR_EMBEDDING(DOC_MODEL USING TO_CHAR(question) AS data)
 WHERE question_vector IS NULL;
COMMIT;
```

**(B) 외부 임베딩 서비스 — `DBMS_VECTOR.UTL_TO_EMBEDDING`**
OCI GenAI 등 외부 임베딩 모델을 자격증명으로 호출합니다(모델 적재 불필요, 크레덴셜 필요).

```sql
UPDATE T_NL2SQL_QUESTION
   SET question_vector = DBMS_VECTOR.UTL_TO_EMBEDDING(
         TO_CHAR(question),
         json('{ "provider":"ocigenai",
                 "credential_name":"OCI_CRED",
                 "url":"https://inference.generativeai.<region>.oci.oraclecloud.com/20231130/actions/embedText",
                 "model":"cohere.embed-multilingual-v3.0" }'))
 WHERE question_vector IS NULL;
COMMIT;
```

> 신규 질문이 등록될 때마다 임베딩을 채우려면, 위 `UPDATE ... WHERE question_vector IS NULL` 을 주기적으로 돌리거나(스케줄러), 애플리케이션에서 INSERT 직후 같은 식으로 채우면 됩니다. **A/B 는 하나만** 선택하고, 검색 시에도 **동일한 모델**로 질의문을 임베딩해야 결과가 일관됩니다.

### 3.1.3 유사 질문 검색

질의문을 같은 모델로 임베딩한 뒤 `VECTOR_DISTANCE`(코사인 거리)로 가장 가까운 질문을 뽑습니다.

```sql
-- (A) DB 내장 모델 기준 — 입력 :q 와 의미가 가까운 질문 상위 10건
SELECT id, TO_CHAR(question) AS question,
       VECTOR_DISTANCE(question_vector,
                       VECTOR_EMBEDDING(DOC_MODEL USING :q AS data),
                       COSINE) AS distance
  FROM T_NL2SQL_QUESTION
 WHERE question_vector IS NOT NULL
 ORDER BY distance
 FETCH FIRST 10 ROWS ONLY;
```

- `distance` 가 **작을수록 유사**합니다(코사인 거리 0 = 동일 방향).
- 대량이면 벡터 인덱스로 근사 검색을 가속할 수 있습니다:
  ```sql
  CREATE VECTOR INDEX IX_NL2SQL_Q_VEC ON T_NL2SQL_QUESTION (question_vector)
    ORGANIZATION INMEMORY NEIGHBOR GRAPH
    DISTANCE COSINE;
  ```
- 필요하면 텍스트 like 검색과 **하이브리드**로 함께 쓸 수 있습니다(정확 일치는 like, 의미 유사는 벡터).

**유사 질문 테이블(`T_NL2SQL_SIMILAR`)로 매칭 → 대표 질문 매핑**
대표 질문마다 여러 표현의 유사 질문을 등록해 두면, 입력이 어떤 표현이든 **가장 가까운 유사 질문**을 찾아 그 **대표 질문(`QUESTION_ID`)** 으로 되돌릴 수 있습니다.

```sql
-- 유사질문 벡터 채우기 (예: DB 내장 모델)
UPDATE T_NL2SQL_SIMILAR
   SET similar_vector = VECTOR_EMBEDDING(DOC_MODEL USING TO_CHAR(similar_question) AS data)
 WHERE similar_vector IS NULL;
COMMIT;

-- 입력 :q 와 가장 가까운 유사질문 → 대표 질문으로 매핑
SELECT q.id            AS question_id,
       TO_CHAR(q.question)       AS question,
       TO_CHAR(s.similar_question) AS matched_similar,
       VECTOR_DISTANCE(s.similar_vector,
                       VECTOR_EMBEDDING(DOC_MODEL USING :q AS data),
                       COSINE) AS distance
  FROM T_NL2SQL_SIMILAR s
  JOIN T_NL2SQL_QUESTION q ON q.id = s.question_id
 WHERE s.similar_vector IS NOT NULL
 ORDER BY distance
 FETCH FIRST 5 ROWS ONLY;
```

- 대표 질문 자체의 `QUESTION_VECTOR` 검색과 **UNION** 해 함께 후보로 삼을 수도 있습니다(대표 + 모든 유사표현을 한 번에 매칭).
- 유사질문도 대량이면 같은 방식으로 벡터 인덱스를 만들 수 있습니다: `CREATE VECTOR INDEX IX_NL2SQL_S_VEC ON T_NL2SQL_SIMILAR (similar_vector) ORGANIZATION INMEMORY NEIGHBOR GRAPH DISTANCE COSINE;`

> **권한/전제**: `VECTOR` 타입·`VECTOR_EMBEDDING`·`DBMS_VECTOR` 는 **Oracle 23ai** 기능입니다. `LOAD_ONNX_MODEL` 에는 모델 적재 권한과 ONNX 모델 파일이, 외부 서비스 방식에는 자격증명(`DBMS_VECTOR.CREATE_CREDENTIAL`)이 필요합니다.

## 4. 참고 · 주의

- **권한**: DB 방식의 테이블 자동 생성에는 접속 계정의 **`CREATE TABLE`** 권한이 필요합니다. 없으면 화면에 오류가 표시됩니다.
- **공유 범위**: 질문·컬럼은 **그 DB에 접속하는 모든 사용자**가 공유합니다. 헤더에서 DB를 바꾸면 그 DB의 목록으로 전환됩니다(질문/컬럼은 각 DB 테이블에 저장되므로 DB별로 분리).
- **평가 프로파일**: 관련성 평가는 화면 상단 **Chat설정에서 선택한 AI Profile** 을 사용합니다. Profile 이 없으면 평가할 수 없습니다.
- **정렬기준**: local storage 방식에만 있습니다. DB 방식은 질문 + 조회할 컬럼만 관리합니다.
- **개인 저장과 공유의 병행**: 두 방식은 독립적입니다. 상단 라디오로 언제든 전환할 수 있으며, 서로의 데이터에 영향을 주지 않습니다.
